# Navigation and Coverage Algorithms

This document summarizes the runtime algorithms used by:

- `wego/launch/navigation_diff_launch.py`
- `wego/scripts/coverage_path_planner.py`

The system is built around a ROS 2 Humble differential-drive LIMO platform. `navigation_diff_launch.py` starts the Nav2 localization/navigation stack, while `coverage_path_planner.py` generates and follows a coverage path over a selected map region and coordinates mine handling behavior.

## 1. `navigation_diff_launch.py`

### Purpose

`navigation_diff_launch.py` starts the Nav2 stack for map-based localization and navigation. It loads a saved occupancy map, runs AMCL localization, starts Nav2 planner/controller servers, and opens RViz with the navigation display configuration.

### Main Components

The launch file starts these major components:

| Component | Role |
| --- | --- |
| `nav2_container` | Composable node container for Nav2 lifecycle components |
| `localization_launch.py` | Starts map server and AMCL localization |
| `navigation_only_launch.py` | Starts Nav2 planner, controller, behavior tree navigator, costmaps, smoother, behavior server, etc. |
| `rviz2` | Visualization using `wego/rviz/navigation.rviz` |

### Inputs

| Input | Description |
| --- | --- |
| `/map` | Saved occupancy grid loaded from the map YAML file |
| `/scan` | 2D LiDAR data used by AMCL and costmaps |
| `/odom` | Odometry from LIMO base |
| `/tf`, `/tf_static` | Transform tree, including `map -> odom -> base_link -> laser_link` |
| `/initialpose` | Initial pose from RViz 2D Pose Estimate |
| `/goal_pose` or Nav2 action goals | Navigation goals |

### Outputs

| Output | Description |
| --- | --- |
| `/amcl_pose` | Estimated robot pose in the `map` frame |
| `map -> odom` TF | AMCL localization transform |
| `/cmd_vel` | Velocity command generated by Nav2 controller |
| Costmap topics | Local/global costmap visualization and planning data |

### Localization Algorithm

Localization is handled by AMCL:

1. Load the saved occupancy grid map.
2. Receive LiDAR scans from `/scan`.
3. Use the differential-drive motion model with odometry prediction.
4. Match LiDAR scans against the map using a likelihood field laser model.
5. Update the particle filter and publish the robot pose.
6. Broadcast the `map -> odom` transform.

Important AMCL settings:

```yaml
robot_model_type: nav2_amcl::DifferentialMotionModel
laser_model_type: likelihood_field
min_particles: 500
max_particles: 2000
update_min_d: 0.05
update_min_a: 0.05
transform_tolerance: 1.0
```

### Navigation Algorithm

Nav2 uses the standard planner-controller pipeline:

1. A goal is given in the `map` frame.
2. The global planner computes a path on the global costmap.
3. The controller follows the path using the local costmap.
4. The behavior tree handles recovery behaviors if needed.
5. The velocity smoother limits acceleration and publishes velocity commands.

Configured planner:

```yaml
planner_server:
  GridBased:
    plugin: nav2_navfn_planner/NavfnPlanner
    use_astar: false
    allow_unknown: true
```

Configured controller:

```yaml
controller_server:
  FollowPath:
    plugin: dwb_core::DWBLocalPlanner
    max_vel_x: 0.26
    max_vel_theta: 1.0
```

### Costmap Algorithm

Both local and global costmaps use:

- Static map layer
- Obstacle layer from `/scan`
- Inflation layer around obstacles

Local costmap:

```yaml
global_frame: odom
rolling_window: true
width: 2
height: 2
resolution: 0.05
```

Global costmap:

```yaml
global_frame: map
track_unknown_space: true
resolution: 0.05
```

The obstacle layer marks and clears obstacles using LiDAR:

```yaml
observation_sources: scan
scan:
  topic: /scan
  obstacle_max_range: 1.0
  raytrace_max_range: 1.5
```

## 2. `coverage_path_planner.py`

### Purpose

`coverage_path_planner.py` performs complete coverage path planning for the selected rectangular map region. It divides the region into grid cells, generates a boustrophedon-style path, follows the path using a pure-pursuit-like controller, handles metal sensor events, coordinates the manipulator, and returns to the safe zone when needed.

### Main Topics and Services

Inputs:

| Input | Description |
| --- | --- |
| `/map` | Occupancy map used for grid generation |
| `/tf` | Robot pose lookup from `map` to `base_link` |
| `/scan` | Front obstacle stop check |
| `/arduino/events` | Metal sensor event strings |
| `/metal_sensor/any_detected` | Boolean metal sensor state |
| `/clicked_point` | Optional RViz clicked corner points |

Outputs:

| Output | Description |
| --- | --- |
| `/cmd_vel` | Direct differential-drive velocity command |
| `/grid_occupancy` | Generated coverage grid |
| `/coverage_path` | Planned coverage path |
| `/coverage_grid_markers` | Grid visualization markers |
| `/coverage_priority_markers` | Visit order visualization markers |
| `/coverage_mission_status` | Text mission status |

Services used:

| Service | Description |
| --- | --- |
| `/start_coverage` | Starts coverage execution |
| `/stop_coverage` | Stops coverage execution |
| `/run_metal_grasp_sequence` | Runs blower and arm pickup sequence |
| `/drop_mine_at_safe_zone` | Drops carried mine at the safe zone |

### Grid Generation Algorithm

The planner supports multiple corner modes. The current configuration uses fixed parameter corners:

```yaml
corner_mode: param
corner_points: '-0.247,-1.621,0.972,-1.471,0.739,0.325,-0.502,0.197'
```

Algorithm:

1. Read four corner points in the `map` frame.
2. Sort the points around their center.
3. Find the longest edge and use its direction as the grid `u` axis.
4. Define the perpendicular direction as the grid `v` axis.
5. Project all corners into this rotated local coordinate frame.
6. Apply `coverage_boundary_margin`.
7. Divide the resulting rectangle by `coverage_cell_size`.
8. For each cell center, check whether it is free in `/map` if `require_free_cell_center` is true.
9. Mark each cell as:
   - `UNVISITED = 0`
   - `VISITED = 50`
   - `BLOCKED = 100`

Current important settings:

```yaml
coverage_cell_size: 0.3
coverage_boundary_margin: 0.15
require_free_cell_center: true
```

### Coverage Path Generation

The main coverage strategy is `cell_centers`.

```yaml
path_strategy: cell_centers
stripe_axis: x
```

The algorithm creates a boustrophedon path, also called a lawnmower or ㄷ-shaped sweep:

1. Iterate over grid rows.
2. On even rows, visit cells from left to right.
3. On odd rows, visit cells from right to left.
4. Skip blocked cells.
5. Store the resulting cell sequence as `coverage_path`.

For `stripe_axis: x`, the pattern is:

```text
row 0: 0 -> 1 -> 2 -> 3
row 1: 7 -> 6 -> 5 -> 4
row 2: 8 -> 9 -> 10 -> 11
```

This reduces unnecessary turnarounds and supports continuous coverage.

### Path Following Algorithm

The current execution mode is:

```yaml
execution_mode: path_follower
```

This means the node follows the generated coverage path directly with `/cmd_vel`, instead of sending each cell to Nav2.

The follower uses a differential-drive pure-pursuit-style controller:

1. Look up current robot pose using TF.
2. Select the active target cell.
3. If the robot is within tolerance, mark the target as reached.
4. If the robot heading differs too much from the path direction, rotate in place first.
5. Choose a lookahead point along the interpolated path.
6. Transform the lookahead point into the robot frame.
7. Compute curvature:

```text
curvature = 2 * y_local / lookahead_distance^2
```

8. Publish:

```text
linear.x = linear_speed
angular.z = clamp(linear_speed * curvature, -max_angular_speed, max_angular_speed)
```

Important parameters:

```yaml
linear_speed: 0.08
lookahead_distance: 0.13
max_angular_speed: 0.25
yaw_tolerance: 0.1
rotate_to_path_yaw: 0.35
```

### Goal Tolerance

The planner uses different tolerances depending on the path context:

```yaml
straight_goal_tolerance: 0.07
corner_goal_tolerance: 0.05
return_goal_tolerance: 0.08
```

Straight sections are allowed to be slightly looser, while corners are stricter so the robot turns from a better pose.

### Reverse Recovery

If the robot overshoots a target and the target lies behind the robot, the planner can reverse slowly toward the target:

```yaml
enable_reverse_goal_recovery: true
reverse_speed: 0.04
reverse_goal_recovery_distance: 0.25
```

This prevents the robot from repeatedly trying to turn forward into a nearby missed point.

### Obstacle Stop

The node checks `/scan` for obstacles in front of the robot.

```yaml
enable_obstacle_stop: true
front_obstacle_stop_distance: 0.12
front_obstacle_angle: 0.8
```

If a scan point is inside the front angular window and closer than the stop distance, the planner publishes zero velocity.

### Safe Zone Policy

The safe zone is the first grid path index:

```yaml
safe_zone_path_index: 1
safe_zone_point: '0.0185,-1.298'
start_direction_mode: safe_zone_nearest_corner
```

The path order is selected so that coverage starts from the corner closest to `safe_zone_point`.

On start:

1. If `reset_to_safe_zone_on_start` is true, `current_index` starts at the safe zone index.
2. If `require_start_in_safe_zone` is true, the robot must be physically near the safe zone before starting.

On complete:

```yaml
return_to_safe_zone_on_complete: true
```

The robot returns to the safe zone after all cells are covered.

### Metal Detection and Mine Handling

Metal detection can come from:

```yaml
metal_events_topic: /arduino/events
metal_state_topic: /metal_sensor/any_detected
mine_detection_mode: sensor
```

When metal is detected:

1. Stop the robot.
2. Determine the current grid cell.
3. Mark that cell as `VISITED`.
4. Add the cell to `mine_cells`.
5. Reverse by `metal_backup_distance`.
6. Call `/run_metal_grasp_sequence`.
7. If the arm sequence succeeds, return to the safe zone.
8. Align to `safe_zone_drop_yaw_rad`.
9. Call `/drop_mine_at_safe_zone`.
10. If drop succeeds, remove the original cell from `mine_cells`, making it traversable again.
11. Resume coverage from the next grid.

Current settings:

```yaml
metal_backup_distance: 0.08
metal_backup_speed: 0.03
return_to_safe_zone_on_mine: true
safe_zone_drop_yaw_rad: 1.5708
```

### A* Return and Resume Algorithm

After mine pickup, the robot does not use the ㄷ-shaped coverage path to return. It uses A* over grid cells.

The A* graph:

- Node: grid cell `(gx, gy)`
- Edges: 4-neighbor movement only
- Cost: 1 per move
- Heuristic: Manhattan distance

Traversability rule:

```text
start cell: allowed
goal cell: allowed
mine cell: blocked unless it is the start cell
other cells: allowed only if VISITED
```

This means the robot avoids unexplored cells while carrying a mine. Unvisited cells are treated as unknown/risky, so return paths are planned only through already confirmed safe cells.

After the mine is dropped successfully, the carried mine cell is cleared:

```text
mine_cells.remove(cleared_cell)
cell state remains VISITED
```

Therefore, that grid becomes safe and can be used in later A* paths.

### Mission State Machine

The coverage planner uses a simple state machine:

| State | Meaning |
| --- | --- |
| `coverage` | Follow the boustrophedon coverage path |
| `return_after_mine` | Return to safe zone after pickup |
| `align_after_mine` | Rotate to configured safe-zone yaw |
| `drop_after_mine` | Call arm drop service |
| `resume_after_mine` | Move from safe zone to next coverage cell using A* |
| `return_complete` | Return to safe zone after all coverage is complete |

### High-Level Coverage Flow

```text
Receive map
→ Generate grid from fixed corners
→ Generate ㄷ-shaped coverage path
→ Wait for /start_coverage
→ Follow path using pure-pursuit controller
→ Mark reached cells as VISITED
→ If metal detected:
    stop
    backup
    run arm pickup sequence
    A* return to safe zone through VISITED cells
    align yaw
    drop mine
    clear mine cell
    A* resume to next grid
→ If all cells visited:
    A* return to safe zone
    stop mission
```

## 3. Relationship Between Nav2 and Coverage Planner

`navigation_diff_launch.py` provides:

- Map server
- AMCL localization
- TF frame consistency
- RViz visualization
- Optional Nav2 navigation stack

`coverage_path_planner.py` currently performs the actual coverage driving itself:

- It uses TF from Nav2/AMCL to know the robot pose.
- It publishes direct `/cmd_vel`.
- It does not rely on Nav2 goals when `execution_mode: path_follower`.

Therefore, in the current configuration, Nav2 is most important for localization and map infrastructure, while coverage driving is controlled by the custom planner.
